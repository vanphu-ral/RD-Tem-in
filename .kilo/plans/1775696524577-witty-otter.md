# Plan: Fix ItemData Datasource Conflict

## Goal

Resolve the "Invalid object name 'ItemData'" error by fixing datasource configuration conflicts between Partner4 (DWH) and Partner6 (WorkZone), ensuring ItemData uses the correct DWH database.

## Analysis

- Both Partner4 and Partner6 configs scan overlapping packages (`com.mycompany.wh.domain` and `com.mycompany.wh.repository`), causing Spring to potentially use the wrong EntityManager.
- ItemDataService uses `@Transactional("partner4TransactionManager")`, but overlaps allow ambiguity.
- Partner6 has no active usage in code, so it can be disabled.
- ItemDataRepository needs isolation in a subpackage to prevent scans from other configs.

## Steps

1. **Disable Partner6 in application-dev.yml**:
    - Add `enabled: false` to `partner6` section to prevent conflicts.

2. **Create Subpackage for ItemDataRepository**:
    - Move `ItemDataRepository.java` from `com.mycompany.wh.repository` to `com.mycompany.wh.repository.partner4`.
    - Update package declaration and any imports.

3. **Update Partner4DatabaseConfiguration.java**:
    - Change `@EnableJpaRepositories` basePackages to `"com.mycompany.wh.repository.partner4"`.
    - Optionally, create subpackage for ItemData entity (`com.mycompany.wh.domain.partner4`) and update `@EntityScan` and entity factory packages.

4. **Create ItemData Table in DWH** (if not exists):
    - Use DDL script to create `dbo.ItemData` with required columns.
    - Populate with data from source (e.g., SAP B1).

5. **Test**:
    - Restart application.
    - Call API and verify no errors.

## Risks

- Disabling Partner6 may affect other features if it's used implicitly.
- Moving files requires careful import updates to avoid compilation errors.
- Table creation assumes DWH is the correct database; if data is in WorkZone, adjust datasource accordingly.

## Dependencies

- No external dependencies.
- Requires database access for table creation.
